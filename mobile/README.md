# Sports Manager · Coach Mobile (PWA)

Mobile attendance for coaches and MICs. Pure web app — no install, opens in
any phone browser. Each school points it at their own Supabase project via
a setup QR generated by the desktop app.

## Quick start (development)

Requires **Node.js 18+** ([download](https://nodejs.org)).

```bash
cd mobile
npm install
npm run dev
```

Vite will print two URLs — `localhost:5173` for desktop browser testing and a
LAN URL like `http://192.168.x.x:5173` you can open on your phone (must be
on the same WiFi).

The app loads with no school configured. To bind it during dev, append the
URL params manually:

```
http://localhost:5173/#/setup?url=https://abcd.supabase.co&anon=eyJ...
```

…or generate a setup QR from the desktop app: Settings → Cloud → "Generate
Mobile Setup QR" → scan it with your phone.

## Production build

```bash
npm run build
```

Output lands in `dist/` — a static SPA. Deploy to Vercel, Netlify, or any
static host. No server-side code, no environment variables at build time
(per-school config lives in `localStorage`).

## Deploying to Vercel

The repo includes a [`vercel.json`](./vercel.json) with the right cache
headers + security headers baked in. Two ways:

**Drag-and-drop (no Git needed):**

1. `npm run build` locally.
2. [vercel.com](https://vercel.com) → **Add New → Project → Deploy without Git**.
3. Drag the `dist/` folder into the upload area → **Deploy**.

**Vercel CLI (one-line redeploys):**

```bash
npm install -g vercel
vercel              # interactive first-time setup
vercel --prod       # redeploy to production
```

After deployment, copy your `.vercel.app` URL and set `PWA_BASE_URL` in the
desktop app's `.env` so generated setup QRs point at it. See
[CLOUD_DEPLOYMENT.md](../CLOUD_DEPLOYMENT.md) Part 3 for the end-to-end flow.

## Architecture notes

- **Router:** HashRouter. Works on any static host without rewrite rules — every URL serves `index.html` and the `#/...` fragment determines the page.
- **Config:** per-school Supabase URL + anon key live in `localStorage` (set by `/setup` from the QR's query params). No build-time env vars — one deployed PWA serves any number of schools.
- **Supabase client:** lazy-initialised on first use after config is set; reset on sign-out / re-setup.
- **RLS does the auth:** the anon key has no privileges of its own. The user's JWT carries `app_metadata: { role, coach_id, mic_id }`, and the cloud's RLS policies scope every read/write accordingly.
- **Service worker:** caches the app shell so the PWA opens offline. Data calls always go straight to Supabase — offline writes are a v2 feature.
